### 📑 Protocol Layer
The protocol layer is the orchestration heart of the system. It handles message framing, request/response linking, and the high-level patterns that ensure messages reach their intended destination.

**Key Classes Include:**
* **Protocol:** The abstract foundation for message exchange.
* **Client:** The initiator of requests and manager of session state.
* **Server:** The provider of capabilities and responder to requests.

---

### 🚚 Transport Layer
The transport layer handles the physical movement of data between clients and servers. While the client often selects a transport automatically, manual configuration allows for granular control over environment variables, authentication, and session management.

#### 1. STDIO Transport (Local Integration)
Designed for local subprocesses and CLI tools. The client launches the server as a subprocess, communicating via `stdin` and `stdout`.

* **Communication:** JSON-RPC messages are newline-delimited (no embedded newlines allowed).
* **Logging:** The server writes UTF-8 strings to `stderr` for logs, ensuring they don't interfere with the `stdout` data stream.
* **Management:** The client manages the server’s entire lifecycle—starting, stopping, and restarting the process as needed.

#### 2. Streamable HTTP Transport (Web & Remote)
The recommended transport for web deployments, providing efficient bidirectional streaming. Unlike STDIO, the server operates as an independent process capable of handling multiple concurrent clients.



* **Communication Flow:** Uses **HTTP POST** for client-to-server messages and **HTTP GET** (typically via Server-Sent Events) for server-to-client streaming.
* **Deployment:** Can be initiated by setting the transport to `http` in the `run()` method (defaulting to `127.0.0.1:8000`).
* **Security Requirements:**
    * **Origin Validation:** Servers must validate headers to prevent DNS rebinding.
    * **Binding:** Local servers should bind to `127.0.0.1` rather than `0.0.0.0` for security.
* **State & Resumability:**
    * **Sessions:** Supports stateful connections via Session IDs assigned during initialization.
    * **Reliability:** Both parties must support message redelivery and connection resumption to handle network instability.

> **Note:** A server must never "broadcast" a message across multiple SSE streams; each JSON-RPC message must be sent on exactly one connected stream.

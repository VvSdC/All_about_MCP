# 🛡️ Best Practices & Security in MCP

To build a production-ready **Model Context Protocol** implementation, you must balance efficiency with a "Zero Trust" security mindset. Below are the core guidelines for building robust and secure servers.

---

## 🛠️ Implementation Best Practices

### 1. Transport Selection
Choose the "pipe" that fits your environment:
* **Local (STDIO):** Use this for desktop apps or local IDE plugins. It’s fast and managed automatically by the host process.
* **Remote (Streamable HTTP):** Use this for cloud services. Always implement **OAuth2** or **API Keys** for authentication.

### 2. Smart Message Handling
* **Input Validation:** Don't trust the LLM. Use **Pydantic** (in Python) or **Zod** (in TypeScript) to strictly validate tool arguments.
* **Progress Reporting:** For long tasks (like searching a massive database), use `progress tokens` so the user doesn't think the AI has frozen.
* **Timeouts:** Never let a tool run indefinitely. Set a hard limit (e.g., 30s) to free up resources.

### 3. Debugging & Monitoring
* **Logging:** Always log message flow and errors to `stderr` (for STDIO) or a dedicated logging service.
* **Health Checks:** Implement a ping/pong or status tool to monitor if your server is still alive.

---

## 🔐 Security & Risk Mitigation

Building an MCP server means giving an AI "hands" to touch your data. Here is how to keep those hands in check:

### 1. The Confused Deputy Problem
This happens when a "Proxy" server is tricked into using its permissions to do something malicious for a third party.
* **Fix:** Never use a single static ID for all users. Force the proxy to get **explicit user consent** for every new client registration.

### 2. The "Token Passthrough" Taboo
**Strict Rule:** Never accept a token from a client and blindly pass it to another service.
* **The Risk:** If a client gives you a stolen token, you become an accomplice in bypassing rate limits or security filters.
* **Fix:** Only accept tokens specifically issued for **your** server. Validate them locally before doing any work.

### 3. Session Hijacking & Prompt Injection
Attackers may try to steal a `Session-ID` to pretend to be a valid user or use "Prompt Injection" to trick the tool into ignoring security rules.
* **Fix:** Use cryptographically secure, random session IDs.
* **Rotation:** Rotate and expire IDs frequently.
* **Sanitization:** Treat every string from the LLM as potentially malicious code.

---

## 📝 Security Checklist

| Category | Best Practice |
| :--- | :--- |
| **Transport** | Use **TLS/HTTPS** for all remote connections. |
| **Binding** | Bind local servers to `127.0.0.1` only. |
| **Data** | Never leak stack traces or internal paths in Error messages. |
| **Limits** | Implement request size limits to prevent Denial of Service (DoS). |

### 🧠 Core Concepts of MCP

The Model Context Protocol is built on a flexible, extensible, and capability-based architecture. This allows servers and clients to evolve independently while ensuring seamless communication via a negotiation system where both parties explicitly declare their supported features during initialization.

The core concepts that drive this ecosystem include:

* **Resources:** Expose raw data and content (files, logs, DBs) to the LLM.
* **Prompts:** Reusable templates and workflows that guide LLM behavior.
* **Tools:** Executable functionalities that allow the LLM to perform real-world actions (API calls, code execution).
* **Sampling:** Allows servers to request LLM completions through the client, enabling agentic loops without giving the server direct model access.
* **Roots:** Define the filesystem or URI boundaries where servers are permitted to operate.
* **Elicitation:** Enables servers to ask users for missing information or clarifications during a task.
* **Transports:** The underlying communication channels (STDIO or HTTP/SSE) that carry the protocol messages.

### 🖥️ Hosts

Hosts are LLM applications (like an AI agent, Claude Desktop, or IDEs) that initiate connections. The host process acts as the container and coordinator.

* **Management:** Creates and manages multiple client instances.
* **Control:** Controls client connection permissions and lifecycle.
* **Security:** Enforces security policies and consent requirements.
* **Authorization:** Handles user authorization decisions.
* **Integration:** Coordinates AI/LLM integration and sampling.
* **Context:** Manages context aggregation across clients.

Host applications handle complex orchestration responsibilities to ensure a seamless user experience.

---

### 🔌 Clients

Each MCP client is created by the host and maintains an isolated server connection.

* **Stateful Sessions:** Establishes one stateful session per server.
* **Negotiation:** Handles protocol negotiation and capability exchange.
* **Routing:** Routes protocol messages bidirectionally.
* **Subscriptions:** Manages subscriptions and notifications.
* **Security Boundaries:** Maintains strict isolation between different servers.

A host application creates and manages multiple clients, with each client having a 1:1 relationship with a particular server. Clients declare capabilities like sampling support and notification handling. Components like **MCPAgents** use the MCP client to access servers, providing the necessary tools, resources, and prompts.

> **Summary:** The **Host** is the application users interact with, while **Clients** are the protocol-level components that enable and manage server connections.

### 🧠 Context
The `Context` object is automatically injected into tool and resource functions that request it via type hints (`ctx`). It serves as the primary interface for a function to interact with the broader MCP environment.

* **Capabilities:** Provides access to the `fastmcp` server instance, client session, request context, sampling, and elicitation.
* **Implementation:** To use context in a tool or resource function, simply add a parameter with the context type annotation:
    ```python
    @mcp.tool()
    def my_tool(name: str, ctx: Context):
        # Accessing server or session info via ctx
        ...
    ```

### 🤖 MCP Agent
An **MCP Agent** is an AI agent that utilizes the Model Context Protocol to interact with external tools and data sources. It bridges the gap between the LLM's reasoning and real-world actions.

* **Unified Interface:** Provides a standard way to use MCP tools with various LLM providers (e.g., through LangChain's agent framework).
* **Customization:** Supports customizable system prompts and conversation memory.
* **Dynamic Chaining:** Instead of hard-coded logic, agents send structured requests to any MCP-compatible tool, receive real-time responses, and chain multiple tools together.
* **Model Requirement:** The LLMs used in MCP agents **must support tool binding** to effectively call functions and handle the resulting data.

---

### ⚙️ Servers
Servers are the "capability providers" of the MCP ecosystem. They focus on specific, well-defined tasks and provide specialized context to the LLM.

**Key Characteristics:**
* **Independence:** Each server provides focused functionality in isolation; multiple servers can be combined seamlessly by the host.
* **Security & Privacy:** Servers receive only the necessary contextual information; the full conversation history stays with the host.
* **Capability Advertisement:** Features (like resource subscriptions or tool support) must be explicitly "advertised" in the server's capabilities during negotiation.

#### MCP Primitives & Control Hierarchy
Servers provide three fundamental building blocks to language models:

| Primitive | Control | Description | Example |
| :--- | :--- | :--- | :--- |
| **Prompts** | **User-controlled** | Pre-defined templates or instructions guided by user choice. | Slash commands, menu options |
| **Resources** | **App-controlled** | Structured data (files, DBs) attached and managed by the client. | File contents, Git history |
| **Tools** | **Model-controlled** | Executable functions that allow LLMs to take real-world actions. | API POST requests, file writing |

### Tool definition

Each tool is defined with name, a human-readable description which the LLM uses to select the appropriate tool to solve the current task and a set of tool-specific parameters in the form of JSON-schema. A tool definition might also include a set of annotations for advance settings.

Tools in FastMCP (from mcp.server.fastmcp) transform regular python functions into capabilities that LLMs can invoke during conversations. When an LLM decides to use a tool:

1. It sends a request with parameters based on tool's schema.
2. FastMCP validates these parameters against the function's signature.
3. The function executes with validated inputs
4. The result is returned to the LLM, which can use it in its response.

This allows the LLM to perform tasks like querying databases, calling APIs, making calculations, or accessing files - extending their capabilities beyond what is in their training data.


#### Decorator (Tool Annotation)

The decorator @<mcp_server_instance_name>.tool (tool annotations) is mandatory to define a tool in mcp server. When a tool is registered, MCP automatically:

- Uses the function name (add) as tool name
- Uses the function's docstring as tool description
- Generates an input schema based on function's parameters and type annotations.
- Handles parameter validation and error reporting


### Tool definition structure (Decorator arguments)

While FastMCP infers the name and description from the function, one can override these and add additional metadata by using arguments in the @mcp.tool decorator. A tool definition includes

- Name: Unique identifier for the tool, sets the explicit tool name exposed via MCP. If not provided, uses the function name.
- Title: Optional human-readable name of the tool for display purposes.
- Description: Human-readable description of functionality. If set, the function's docstring is ignored for this purpose
- Annotations: Optional ToolAnnotations objects or dictionary to add additional metadata about the tool, which describes the tool behavior.

Example using FastMCP:

```
@<mcp_server_instance_name>.tool(
    name: string,                   // Unique identifier for the tool (optional)
    title: string,                  // Title of the tool (optional)
    description: string,            // Human-readable description (optional) 
    annotations: {                  // Optional hints about tool behavior (optional)
        'title': string;            // Human readable-title for the tool (optional)
        'readOnlyHint': boolean;    // If true, the tool doesn't modify its environment (optional)
        'destructiveHint':boolean;  // If true, the tool may perform destructive updates (optional)
        'idempotentHint': boolean;  // If true, repeated calls with same args have no additional impact (optional)
        'openWorldHint': boolean;   // If true, tool interacts with external entities (optional)
    },
    structured_output: boolean      // Optional
)

```

Security is the major concern for tools providing access to internal sensitive data or interacting with external systems for various external-API based functionalities. We should implement authentication, and proper appropriate authorization checks where needed. Tool implementation must validate all parameters against the schema, validate URLs and external identifiers, and sanitize file paths and system commands, and most importantly prevent command injections.


### Tool annotations

Tool annotations offer extra information (metadata) about how a tool works, guiding clients on its display and management. These annotations communicate how tools behave to client applications without consuming token context in LLM prompts. Annotations server several purposes in client applications:

- Adding user-friendly titles for display purposes
- Indicating whether tools modify data or systems.
- Describing the safety profile of tools (Destructive vs non-destructive)
- Signalling if tools interact with external systems.

With these annotations hint at tool's nature and impact, they should not be used for security decisions. Tool annotations provide UX-specific information without affecting model context, which helps the clients categorize and present tools appropriately.

The MCP specification defines the following annotations for tools:

Annotations     Type        Default         Description
Title           string      -           A human-readable title for the tool, useful for UI display
readOnlyHint    boolean     false       If true, indicates the tool does not modify its environment
destructiveHint boolean     true        If true, the tool may perform destructive updates (only meaningful when readOnyHint is false)
idempotentHint  boolean     false       If true, calling the tool repeatedly with sam arguments has no additional effect (Only meaningful when readOnlyHint is false)
openWorldHint   boolean     true        If true, the tool may interact with an "open world" of external entities
